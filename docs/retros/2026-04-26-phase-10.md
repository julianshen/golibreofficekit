# Phase 10 retro — TDD discipline broke

**Date:** 2026-04-26
**PR:** [#22 — feat(lok): Phase 10 — command values & window events](https://github.com/julianshen/golibreofficekit/pull/22)
**Outcome:** merged after a multi-agent review found bugs that red-first tests
would have caught at write time.

## What happened

Phase 10 shipped on `feat/command-values` in two commits:

| Commit    | Subject                                                   |
|-----------|-----------------------------------------------------------|
| `8b05f0d` | feat(lok): Phase 10 — command values & window events      |
| `966d0ff` | test(lok): Phase 10 — unit tests for command values, …    |

The implementation commit's own message admitted the violation:
> _New code not yet tested (per plan)._

`CLAUDE.md` rule 3 ("Strict TDD … do not batch 'I'll add tests later'") was in
force the entire time. The rule was acknowledged and ignored.

## Root cause

The plan (`docs/superpowers/plans/2026-04-26-phase-10-command-values.md`) was
sliced by **layer**, not by **method**:

- Tasks 1–7: implement layered surface (lokc shims → backend → public methods)
- Task 8: add unit tests for everything in 1–7
- Task 9: integration tests
- Task 10: review

The agent followed the plan faithfully. Following the plan meant building the
whole stack, then bolting tests on. That is by definition not TDD; the plan
structure made the anti-pattern the path of least resistance.

The `CLAUDE.md` TDD rule named the batch-then-test failure mode but did not
constrain the *plan shape*, so the plan was free to encode it.

## Bugs the missing red tests let through

These were caught only in PR review, not at write time. With per-method TDD
each one would have surfaced as a failing test before the implementation was
written.

1. **`mapLokErr` bypass.** All 13 Phase 10 forwarders in `lok/real_backend.go`
   returned `lokc.ErrUnsupported` raw instead of translating to the public
   `lok.ErrUnsupported`. The `fakeBackend` already returned the public sentinel,
   so unit tests never saw the gap; only review against the existing Phase 8/9
   pattern caught it.
2. **`int64 → int` truncation in `PostWindowGestureEvent`.** Phase 7's
   `PostMouseEvent` validates with `requireInt32XY`; the gesture variant did
   not validate `x`, `y`, or `offset` at all. A red test asserting "values
   above `math.MaxInt32` produce a `*LOKError`" would have driven the validator
   into existence.
3. **Forms tests asserted nothing useful.** `TestSendContentControlEvent` and
   `TestSendFormFieldEvent` only checked the error return; the `fakeBackend`
   ignored `argsJSON`. A regression that dropped the parameter would have
   passed. Writing the test first ("backend receives the JSON I sent") would
   have forced the capture field on `fakeBackend`.
4. **`SendDialogEvent` truncated `uint64 → uint32`** in the test fake. Real
   LOK's `sendDialogEvent` takes `unsigned long long`. A red test
   round-tripping `1<<33` would have caught it.
5. **`internal/lokc/` Phase 10 wrappers had 0% coverage.** The whole package
   sat at 72.7% (vs. the 90% rule) because no test file existed for the new
   wrappers. Existing fake-LOK helpers (`newFakeDoc`) made these trivial to
   write — they just weren't written, because the plan deferred all testing
   to Task 8.
6. **`IsCommandEnabled` doc/code mismatch.** The doc promised an error path
   that the code didn't implement. Writing the doc and the test in the same
   pass — TDD's natural rhythm — would have shown the contradiction.

## Process change

`CLAUDE.md` rule 3 now explicitly forbids the two failure modes:

- A plan task labelled "implement A, B, C" + a separate task "test A, B, C"
  is itself the anti-pattern; restructure the plan before starting.
- A commit titled "feat: implement X" with "tests in next commit" is the
  same anti-pattern at the git layer; each commit lands the test for the
  behaviour it introduces.

For Phase 11, the plan must list one task per method (or per small
co-designed group), and each task lands its test and its impl together.

## What worked

- The four-layer architecture (`lok` → backend → realBackend → `internal/lokc`)
  held up cleanly; no rework needed.
- The multi-agent review (`/pr-review-toolkit:review-pr`) caught all six bugs
  above in a single pass and gave file:line citations. Use it earlier next
  time — before the merge, not after.
- The `fakeBackend` pattern is good infrastructure; the Phase 10 mistake was
  not using it for behavioural assertions, not the pattern itself.
